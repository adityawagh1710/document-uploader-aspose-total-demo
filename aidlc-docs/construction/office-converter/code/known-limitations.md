# Known Limitations of the Generated Code

What runs out-of-the-box vs what requires operator setup.

## Runs without any Aspose SDK present

These tests + checks pass on a clean Python 3.11 environment with the
dev dependencies installed (`uv sync`):

- All unit tests under `tests/unit/` — chunk planner, cache, license XML
  parsing, logging, config, probe format detection, qpdf wrapper (uses
  the real qpdf binary; needs `apt-get install qpdf`).
- All property-based tests under `tests/property/` — same as unit; some
  depend on qpdf being installed.
- Integration tests under `tests/integration/` — use the fake worker
  stand-in from `tests/conftest.py` (a Python script that returns
  pre-canned PDFs). End-to-end Aspose rendering is NOT exercised.
- `mypy --strict office_convert/` and `ruff check .` pass.

## Requires operator Aspose SDK tarball (build time)

The C++ worker source under `worker_cpp/` currently throws
`"SDK not linked in this build"` at runtime because:

- The `formats/*.cpp` files have the Aspose API call examples in
  comments but the actual calls are stubbed out so the C++ compiles
  cleanly even without the SDK headers being present.
- `worker_cpp/CMakeLists.txt` only links against the Aspose libraries
  if `-DASPOSE_SDK=<path>` is passed.

To enable real rendering:

1. Place `aspose-total-cpp.tar.gz` next to the Dockerfile.
2. Edit `formats/docx.cpp`, `formats/pptx.cpp`, `formats/xlsx.cpp`,
   `formats/pdf.cpp` and `probe.cpp` to uncomment the real Aspose API
   calls (the commented blocks are commented `#include <Aspose...>` and
   commented `Aspose::Words::Document(...)` etc.). The structural
   scaffolding is in place.
3. Same for `license.cpp` (currently scaffolded to verify file
   readability only).
4. Build the Docker image: `docker build -t office-convert .`

## Requires operator Aspose license (runtime)

Even after the C++ worker is fully wired:

- Without `/aspose/license.lic` bind-mounted, the worker exits 2 on
  every request and the server returns 503 `license_expired`.
- With an expired `.lic`, same.
- License must be Aspose.Total **C++** scope. The Python-via-.NET temp
  license is NOT compatible with this build.

## Document corpus fixtures

`tests/corpus/_generate.py` produces:

- `simple.pdf` — included (generated locally via ReportLab during code
  generation).
- `small.docx`, `medium.docx` — need `python-docx`; generated by
  `python -m tests.corpus._generate` after `uv sync`.
- `simple.pptx`, `complex.pptx` — need `python-pptx`; same.
- `single_sheet.xlsx`, `multi_sheet.xlsx` — need `openpyxl`; same.

Generator is idempotent: existing files are kept untouched.

## Test fixture limitations

- `tests/integration/test_convert_endpoint.py::test_convert_returns_pdf`
  uses the fake worker and verifies the HTTP shape, not Aspose render
  fidelity. Real conversion correctness must be verified end-to-end
  against the running container (see `tests/e2e/` below).
- The fake worker (in `conftest.py`) writes a synthetic PDF for any
  format, so format-specific Aspose quirks are not exercised in the
  in-process integration suite.

## End-to-end test suite

`tests/e2e/` uses **Testcontainers** to bring up the real Docker image
and exercise `/convert` over real HTTP. Coverage that in-process tests
cannot reach:

- Dockerfile correctness (apt deps, env vars, USER, LD_LIBRARY_PATH)
- Real C++ worker binary loading (Aspose .so symbols resolvable)
- Real Aspose render + license activation
- Real qpdf concat at real sizes
- Real prlimit RLIMIT_AS=2G behavior

**Gating**: skipped unless `OFFICE_CONVERT_E2E_LICENSE` env var points at a
valid Aspose.Total C++ Temporary License `.lic` file. CI without an
Aspose license runs only the in-process suite.

**Requirements**:

- Docker daemon accessible at test time.
- Image built with the real Aspose SDK tarball in the build context
  (`docker build -t office-convert:test .`).
- The `e2e` optional dependency installed: `uv pip install -e .[dev,e2e]`
  (adds `testcontainers` and `httpx`).

**Invocation**:

```bash
OFFICE_CONVERT_E2E_LICENSE=/path/to/license.lic \
OFFICE_CONVERT_E2E_IMAGE=office-convert:test \
pytest tests/e2e -m e2e
```

Without the real Aspose SDK linked into the worker binary, `test_simple_pdf_converts`
and `test_docx_converts_if_corpus_present` may still pass HTTP-shape
assertions but the conversion itself will fail with `render_failed` (the
scaffolded worker throws "SDK not linked"). The tests accommodate this:
they accept either 200 (real Aspose) or 500 with `failure_class: render_failed`
(scaffolded worker), so the e2e suite still validates the Dockerfile and
HTTP plumbing even before Aspose is wired in.

## What v1 explicitly does NOT do (per requirements + stories)

See `aidlc-docs/inception/user-stories/stories.md` section "Explicit
Non-Goals" (NG-1 through NG-9):

1. No application-layer auth.
2. No per-tenant quotas.
3. No `/metrics` endpoint.
4. No automatic cache eviction.
5. No hot-reload of configuration.
6. No HA / replication / failover.
7. No committed SLO.
8. No PDF post-processing (linearization, PDF/A, signing).
9. No non-x86_64 host support.

## Code-level placeholder behaviors

A few places where the generated code is a placeholder pending real
integration:

| Location | Placeholder | Fill in by |
| -------- | ----------- | ---------- |
| `office_convert/server.py` | `X-Chunks-Rendered` etc. are NOT set as response headers (the orchestrator's `ConversionResult` is logged but not pulled into headers due to HTTP-trailer limitations in FastAPI) | Future improvement: switch to a header-pre-compute model with an upper bound or move metadata into the body for a wrapping JSON response style |
| `office_convert/cache.py` | Cache disabled in compose.yaml (code fully functional) | Enable by setting `OFFICE_CONVERT_CACHE_DIR=/cache` + mounting a volume once conversions are verified stable |

The header-metadata gap is the most user-visible v1 limitation; it does
not affect correctness but pipelines that programmatically read these
headers (per US-PD-01's acceptance criteria) need to grep server logs
for now.

## XLSX performance trade-offs (2026-05-15)

High-aesthetics XLSX files (charts, conditional formatting, decorative
shapes, formulas) use render-side optimizations that trade output
fidelity for speed:

| Optimization | Trade-off |
| ------------ | --------- |
| `SetImageResample(150, 80)` | Charts/images at 150 PPI / 80% JPEG instead of full resolution. Screen-quality, not print-quality. |
| `SetOptimizationType(MinimumSize)` | Skips full font embedding for ASCII 32-127; optimizes border rendering. |
| `SetEmbedStandardWindowsFonts(false)` | Standard fonts not embedded — PDF viewer must have them installed. |
| `SetParsingFormulaOnOpen(false)` | Formula results use cached values from the file, not recalculated. If the file was saved without cached results, formula cells may render blank. |

These are applied unconditionally to all XLSX conversions. A future
`options.xlsx_quality: "high" | "fast"` parameter could let callers
opt into full-fidelity rendering at the cost of longer conversion time.
