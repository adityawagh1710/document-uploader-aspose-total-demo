## Generation Approach

**Scaffolding completeness**:

- **Python**: full implementation, fully testable without Aspose
  (subprocess invocations mockable; chunk planner / qpdf wrapper /
  cache / license parser / config / logging are real working code).
- **C++ worker**: structural scaffold — main, argv parser, error
  translation, license wrapper, format dispatch — all written. The
  actual Aspose render/probe calls inside each format file are
  written against documented Aspose.Total C++ API patterns but
  cannot be fully verified without the user's Aspose SDK headers
  and `.so` files in place. Build will succeed only when operator
  provides the Aspose tarball in build context (per NFR Q5).
- **Tests**: unit + PBT fully runnable in any Python environment.
  Integration tests (`test_convert_endpoint.py`) use mocks for the
  Aspose worker subprocess so they run without the real Aspose SDK;
  separate end-to-end smoke testing is performed via Build and Test
  with the real container.
- **Document corpus**: generated fixtures committed once via
  `tests/corpus/_generate.py`. Generator script is the source of
  truth; output files are checked in for hermetic test runs.

**FR/NFR traceability**: each Python module's docstring references
the FR/NFR it satisfies (e.g., `chunk_planner.py` → FR-3, FR-4,
NFR-5, NFR-6).
